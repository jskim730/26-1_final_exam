## The deliverable and submission constraints

The bot you build/iterate on is what gets submitted. These constraints shape every decision:

- Submit **one `main.py`** (the contest requires that exact name). Start by copying `sample-code.py` → `main.py` and filling in `decide()`; the I/O loop, state-tracking, and pathfinding are already correct.
- **Python 3.12.3**, **Ubuntu**, **single CPU core**, **no network access**, **exit code must be 0** (exit cleanly when the referee sends `FINISH`). Code ≤ 1 MiB; optional `data.bin` ≤ 10 MiB (present only at runtime).
- Allowed libraries: `numpy`, `scipy`, `sortedcontainers`, `more-itertools`, `torch` (cpu), `tensorflow`. Standard library otherwise. **No Windows-only APIs** — local dev here is Windows but the judge is Linux.
- TLE is measured by **CPU time**. Per turn there is a 100 ms soft cap with 5 "countdown" tokens of slack (first turn's effective budget is generous, ~600 ms — fine for heavy one-time precompute like all-pairs paths). The local testing tool measures **wall-clock** (`time.monotonic`), while the judge measures **CPU time**, so leave headroom.

## Architecture: how a game runs

**Two-process model.** `testing-tool.py` is the referee. It spawns the LEFT and RIGHT bots as subprocesses and is the *only* holder of authoritative game state. Each bot is fed information and must **reconstruct its own view of the world from the per-turn result block** — there is no shared state. `sample-code.py` demonstrates this: `read_turn_result()` (sample-code.py:232) replays the referee's outputs to keep a local `GameState` in sync.

**Protocol (per bot, over stdio):**
1. `READY <LEFT|RIGHT>` + map (N, K, zone coords, strongholds, adjacency) → bot must print `OK` within 1 s.
2. Each day: referee sends `START TURN T`; bot prints `COMMAND` … (`MOVE`/`TRAIN`/`UPGRADE`, any order) … `END`; referee replies with a result block (`TURN`/`TIME`/`UPGRADE`/`TRAIN`/`MOVE`/`DAMAGE`/`SIEGE`/`END`).
3. `FINISH` → exit immediately and cleanly (do not try to read a result).

**The turn pipeline order is critical and not obvious from the prose rules.** The referee resolves a day in this fixed order (`run_game`, testing-tool.py:1395-1422):
1. Parse + charge gold: **upgrades/builds → move orders (charge only) → train (charge only)**.
2. **Day movement** — moving warriors advance ≤1 zone (`apply_day_movement`).
3. **Spawn trained warriors** — they appear at HQ *after* movement, so a newly trained warrior cannot act the day it is trained (testing-tool.py:1417).
4. **Combat** (`apply_day_combat`).
5. **Siege** (`apply_day_siege`).
6. **Work/income**, then **upkeep/food** (`apply_evening_work`, `apply_evening_upkeep`).

`sample-code.py`'s `read_turn_result()` mirrors this same order. Strategy lives only in `decide()` (sample-code.py:437); all-pairs path costs are precomputed once via Floyd–Warshall in `calculate_paths()` (sample-code.py:364).

## Game mechanics that drive strategy (verified against the simulator)

- **Siege happens only on overkill.** A building loses HP only from attacks that find *no living enemy warrior* in its zone (`apply_day_combat` counts "idle" attacks → `siege_damage`, testing-tool.py:1092). To damage/destroy an HQ you must field enough attackers to clear all defenders *and* have surplus attacks left. A single defending warrior absorbs the surplus and reduces siege to zero.
- **Combat targeting:** each attack hits the enemy warrior with the **lowest HP**, ties broken by **smallest id** (`_damage_tick`, testing-tool.py:1011). A building's **turret attacks warriors first**; only idle turret attacks become siege. Attack counts are fixed at the start of combat (warriors that drop to 0 HP mid-fight still counted).
- **Movement:** one zone per day along the path minimizing the sum of `ceil(euclidean distance)` between zone centers, ties broken by **smaller zone id**. A warrior is **pinned** (cannot leave) if an enemy warrior shares its current zone (testing-tool.py:984).
- **Economy:** income = `15 × min(friendly warriors in zone, building work_cap)`; each warrior eats 2 gold/day or loses 1 HP (`HUNGER`) if unpaid. Balance training/upkeep or starve.
- **Win:** destroy the enemy HQ within 200 days; otherwise the side with higher HQ HP wins (equal HP → draw).

## Important gotcha: trust the code, not the rules text

`게임규칙.txt` and `대회진행규칙.txt` are raw site copy-pastes — **tables and numbers are exploded across many lines** (e.g. `1200` appears as `1`/`200`/`1200개` on separate lines) and are easy to misread. The **authoritative numeric values** live in `HQ_LEVELS` / `BASE_LEVELS` and the module constants of `testing-tool.py` (mirrored in `sample-code.py`). When you need an exact cost/HP/turret value or a behavioral edge case, **read the simulator, not the txt files.**
